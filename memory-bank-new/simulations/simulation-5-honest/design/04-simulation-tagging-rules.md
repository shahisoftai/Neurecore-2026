# Item 4: Simulation Tagging Rules

This document defines how every simulation-generated record is tagged, how the simulation run identity is structured, and how production retrieval queries can filter simulation artifacts in and out.

## 1. The simulationId (globally unique URI)

A `simulationId` is a globally unique content-addressable URI. It is **not** merely a short string; it is a resolvable identifier that can be quoted across systems, logged in audit trails, and referenced by external tooling without ambiguity.

**Format:**

```
sim://<year>/<month>/<day>/<org-slug>/<framework>/<seq>
```

| Component | Example | Purpose |
|---|---|---|
| `sim://` | `sim://` | URI scheme, marks this as a simulation identifier |
| `<year>/<month>/<day>` | `2026/07/17` | The day the run was started (UTC) |
| `<org-slug>` | `neurocore` | The org or tenant that owns the simulation. For multi-org tenants this is the org slug, not the tenant slug. |
| `<framework>` | `sim5` | The framework that produced the records |
| `<seq>` | `000001` | A monotonic zero-padded 6-digit sequence. The first run on a given day for a given org/framework is `000001`. Sequence is per-`<year>/<month>/<day>/<org-slug>/<framework>` tuple, **not** global. |

**Example:** `sim://2026/07/17/neurocore/sim5/000001`

**Why a URI:**

- A URI is unambiguous across systems. Two runs in two orgs on the same day have different URIs.
- The scheme makes it easy to filter log lines and audit trails for simulation artifacts.
- A future "simulation registry" service can index runs by URI.

**Why the per-day sequence (not global):**

- A 6-digit sequence is enough for any realistic daily run count in a single org. A global sequence would require a coordinated counter that adds latency and contention.
- The combination of date + org + framework already disambiguates across days and orgs. The sequence is the per-day tiebreaker.

**Two-id system:**

The framework tracks two distinct identifiers:

| Identifier | Type | Example | Where it lives |
|---|---|---|---|
| `simulationId` | URI (the one above) | `sim://2026/07/17/neurocore/sim5/000001` | The simulation Project's `metadata.simulation.simulationId`. Stored as a string on every record. |
| `simulationRunId` | CUID (database row id) | `cm9x1y2z3a4b5c6d7e8f9g0h1` | The simulation Project's `id` field. Foreign-key target for child records that want a stable pointer. |

The `simulationId` is the human-meaningful identifier; the `simulationRunId` is the database join key. They are stored side by side. The `simulationId` is the value used in `simulationId` columns on all child records; the `simulationRunId` is only used for `@@unique` constraints and joins.

**The `simulationId` is set on every record produced during a simulation:**

- `Project` (the simulation Project itself; both fields)
- `TimelineEvent` (every injected event)
- `ProjectDecision` (every decision)
- `CommunicationThread` (every deliberation, control thread, debate, challenge)
- `HermesMessage` (every message in those threads)
- `KnowledgeEntry` (counterfactuals, learning updates, scorecards, restricted test knowledge)
- `MissionFeedItem` (every nudge)
- `ApprovalRequest` (every approval gate)
- `Task` (every action item)
- `Routine` (every scheduled routine)
- `IdempotencyRecord` (the idempotency records for the simulation's day-run calls)
- `DecisionEvaluation` (every evaluation of a decision; new in revision 2)

A record with a non-null `simulationId` is a simulation artifact. A record with a null `simulationId` is a real production record.

**Validation:** the backend validates the `simulationId` URI shape on every write. Malformed URIs are rejected with `INVALID_SIMULATION_ID`. The sequence component must match `^\d{6}$` and the date components must be real dates.

## 2. The correlation ID

A `correlationId` is a single identifier that ties together all records produced during one logical operation (e.g. "Day 12 run", "the supplier delay cascade", "the audit of decision X"). It is set on the `TimelineEvent` (and propagated to any record linked to the event).

| Component | Example | Purpose |
|---|---|---|
| `corr` | `corr` | Tag |
| `<simulationId>` | `sim://2026/07/17/...` | The run |
| `<step>` | `day-12-supplier-delay` | The specific operation within the run |

Example: `corr-sim://2026/07/17/neurocore/sim5/000001-day-12-supplier-delay`

Within a single day-run, all records created by that day's Reality Engine share one `correlationId`. Within a cascade, all child events share a `correlationId` and a `rootEventId` pointing at the original trigger.

## 3. The trace ID

A `traceId` is a W3C-compatible distributed-tracing identifier (32 hex characters). It is generated by the engine that initiated the operation (e.g. the day-run orchestrator). The `traceId` propagates to every record produced in that operation.

If a downstream system (the agent runtime, the knowledge service) generates its own trace, that goes in `HermesMessage.metadata` or `Task.metadata` rather than overwriting `traceId`.

## 4. The causation ID

A `causationId` is the `id` of the immediate record that caused this one. For a child `TimelineEvent` in a cascade, `causationId` is the parent event's `id`. For a `ProjectDecision` that responds to a `TimelineEvent`, `causationId` is the event's `id`. This is a useful debugging breadcrumb; it is not a foreign key.

## 5. Visibility filtering rules

The simulation framework must not pollute production retrieval. The following filters are enforced at the service layer (not the database level, because `simulationId` is nullable on most tables and we want production queries to keep working unchanged).

### 5.1 Default Knowledge retrieval

The Knowledge service adds a default filter that excludes entries where `metadata.visibilityScope='SIMULATION_ONLY'` or where `metadata.simulationRestricted=true`. The filter is applied unless the caller explicitly passes `includeSimulation: true`.

### 5.2 Default Timeline retrieval

The Timeline service adds a default filter that excludes `TimelineEvent` rows where `category='SIMULATION'` and `simulationId IS NOT NULL`, unless the caller passes `includeSimulation: true`. The events still exist in the database; they are just hidden by default. The simulation overview UI passes `includeSimulation: true`.

### 5.3 Threads and decisions

Threads and decisions tagged with a non-null `simulationId` are still visible in their normal UI, but a small `🧪 Simulation` badge is shown next to the title so a user can tell at a glance.

### 5.4 Mission feed

Mission feed items with a non-null `simulationId` are filtered out of the main user-facing mission feed by default. They appear in the simulation overview drawer instead.

### 5.5 Approvals and tasks

Same as threads: visible everywhere with a badge, but tagged.

## 6. Enforcement points

The visibility filters are enforced in the following service methods. They cannot be bypassed by the simulation framework itself; only an explicit `includeSimulation: true` parameter exposes them.

| Service method | Default filter |
|---|---|
| `KnowledgeService.search()` | Exclude `visibilityScope='SIMULATION_ONLY'` |
| `TimelineService.listEvents()` | Exclude `category='SIMULATION' AND simulationId IS NOT NULL` |
| `MissionFeedService.listForUser()` | Exclude `simulationId IS NOT NULL` |
| `ProjectDecisionService.listForProject()` | No default filter; UI adds a badge |
| `CommunicationThreadService.listForContext()` | No default filter; UI adds a badge |
| `TaskService.listForUser()` | No default filter; UI adds a badge |
| `ApprovalService.listForUser()` | No default filter; UI adds a badge |

The simulation overview UI passes `includeSimulation: true` to each of these.

## 7. Auditability

Every record created by the simulation framework is auditable:

- `createdByUserId` is set when a human user action (e.g. running the day) initiates the engine.
- `createdByAgentId` is set when an agent-runtime call produces the record.
- When neither applies (e.g. a background reconciliation job), `createdByServiceIdentityId` is set to the `ServiceIdentity.id` that performed the action. **A `ServiceIdentity` is not a `User`; it is its own first-class concept** (see `03-timeline-event-schema.md` and `06-authorization-matrix.md`).
- The audit log records the engine, version, and operation.

The audit log is preserved even after the simulation completes and the simulation artifacts are eventually cleaned up.

## 8. Version block (required for deterministic replay)

The simulation Project's `metadata.simulation` includes a `versions` block. This block is the source of truth for whether two runs of the same seed are truly equivalent.

```json
{
  "versions": {
    "framework": "1.0.0",
    "rng": { "algorithm": "xoshiro256**", "version": "1.0.0" },
    "schema": "v1",
    "scoring": "v1",
    "llm": {
      "provider": "openai",
      "providerVersion": "1.0.0",
      "model": "gpt-4o-mini",
      "modelVersion": "2024-07-18",
      "structuredOutputMode": "json_schema"
    },
    "engines": {
      "realityEngine": "1.0.0",
      "decisionEngine": "1.0.0",
      "debateEngine": "1.0.0",
      "devilAdvocate": "1.0.0",
      "approvalOrchestrator": "1.0.0",
      "taskOrchestrator": "1.0.0",
      "auditor": "1.0.0",
      "scoringService": "1.0.0"
    }
  }
}
```

**The framework does not let a day-run proceed if the `versions` block does not match the values recorded when the run was created.** If a user wants to run with newer engine code, they create a new run (new `simulationId`, new `simulationRunId`). They do not reuse an old `simulationId` with new code. This guarantees that any two runs that share a `seed` and a `simulationId` produce the same outcomes.

When a day-run is invoked, the server compares the request's `expectedVersions` against the run's stored `versions`. If they differ:
- The day-run returns 409 `VERSION_MISMATCH`.
- The error body lists which version(s) differ.
- The user must create a new run if they want to use the new versions.

**Recording versions with every evaluation:** every `DecisionEvaluation` and every running-score `KnowledgeEntry` includes the `scoringVersion` field. Every `HermesMessage` produced by an agent-runtime call includes the `provider`, `model`, `modelVersion`, and the agent's `promptVersion`. This is already part of the existing `HermesMessage.metadata`, but the framework now requires these fields to be present on simulation messages.

**The `modelVersion`** is the provider's model version string (e.g. OpenAI's snapshot date). It is required for true reproducibility because the same model name can return different outputs across snapshots.

## 9. Cleanup and retention

Simulation runs can be retained indefinitely (auditability) or cleaned up after a configurable period (default 90 days). The cleanup process is a tenant-admin operation:

1. The admin sets a retention period for the simulation run.
2. After the period elapses, a background job deletes all records with that `simulationId` in cascade, except the `IdempotencyRecord` rows (kept for audit) and the simulation Project itself (kept as a tombstone).
3. The deletion is logged.

The retention period is recorded in the simulation Project's `metadata.simulation.retentionDays`.

## 10. Synthetic-data containment summary

| Synthetic datum | Where it lives | Where it must NOT live |
|---|---|---|
| Hallucination claims | Control thread message with `envelope.kind='synthetic_claim'` | `KnowledgeEntry` (any kind) |
| Hidden-information facts | `KnowledgeEntry` with `metadata.visibilityScope='SIMULATION_ONLY'` | Default Knowledge retrieval |
| Conflicting facts (for contradiction tests) | Restricted entries with `metadata.visibilityScope='SIMULATION_ONLY'` | Default retrieval |
| Test-only agent personas | `Agent` records with `metadata.simulationId=<id>` and `isSelected=false` in production views | Active agent pool used by mission feed |

Each of these is enforced both in the service layer (filters) and in a regression test (an automated check that asserts the synthetic data does not appear in production retrieval).
