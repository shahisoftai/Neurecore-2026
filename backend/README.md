## Description

[Nest](https://github.com/nestjs/nest) framework TypeScript starter repository.

## Project setup

```bash
$ npm install
```

## Compile and run the project

```bash
# development
$ npm run start

# watch mode
$ npm run start:dev

# production mode
$ npm run start:prod
```

## Run tests

```bash
# unit tests
$ npm run test

# e2e tests
$ npm run test:e2e

# test coverage
$ npm run test:cov
```

## Deployment

When you're ready to deploy your NestJS application to production, there are some key steps you can take to ensure it runs as efficiently as possible. Check out the [deployment documentation](https://docs.nestjs.com/deployment) for more information.

If you are looking for a cloud-based platform to deploy your NestJS application, check out [Mau](https://mau.nestjs.com), our official platform for deploying NestJS applications on AWS. Mau makes deployment straightforward and fast, requiring just a few simple steps:

```bash
$ npm install -g @nestjs/mau
$ mau deploy
```

With Mau, you can deploy your application in just a few clicks, allowing you to focus on building features rather than managing infrastructure.

## Run locally in production (Vercel parity)

Follow these steps to run the backend locally with production settings similar to Vercel:

- Install dependencies:

```bash
pnpm install
```

- Copy the example production env and edit secrets:

```bash
cp .env.production.example .env
# edit .env and set DATABASE_URL, REDIS_URL, JWT_SECRET, etc.
```

- Generate Prisma client (if using Prisma) and build the project:

```bash
pnpm prisma generate
pnpm run build
```

- Run production locally (sets NODE_ENV=production):

```bash
pnpm run start:local-prod
```

- Start Postgres + Redis using the included compose file (helper script):

```bash
./scripts/start-local-prod.sh
# or run the docker compose directly from this folder:
# docker compose up -d
```

Vercel smoke deployment

To verify platform routing before promoting changes to production, add a small serverless handler under `backend/api/` (this repo includes `backend/api/index.js` and `backend/api/handler.js`). Then run locally or deploy:

Run locally with Vercel CLI (recommended):

```bash
# from repo root or backend folder
cd backend
vercel dev
# then open http://localhost:3000/api
```

Or deploy a temporary preview to Vercel:

```bash
cd backend
vercel --prod --confirm
# then visit your deployment URL and check /api and /api/handler
```

If `/api` returns the smoke JSON, the platform routes requests correctly. If you still see 404, inspect `vercel.json` rewrites and the deployment logs.

Notes:

- You can also use `vercel dev` at the project root; Vercel will respect `vercel.json` when running locally.
- The repository includes `vercel.json` with production environment defaults — do not commit secrets to that file. Use `vercel env pull` or your own secret manager when testing with real credentials.
